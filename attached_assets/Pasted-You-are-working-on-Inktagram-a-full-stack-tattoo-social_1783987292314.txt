PHASE 1 — SECURITY (blockers)
=========================

1. Password-hash & email disclosure. In server/storage.ts, many queries do `db.select().from(schema.users)` or embed the full `users` row in joins, so `hashedPassword` (and email) leak to clients — including via the PUBLIC `GET /api/search`.
   - Fix at the source: create a reusable list of "public user columns" (id, username, role, firstName, lastName, bio, avatarUrl, bannerImageUrl, website, instagram, tiktok, twitter, location, isVerified, verificationStatus, createdAt) and use explicit column selection in EVERY query that returns user data to clients: searchUsers (~line 577), searchPosts author (~590), getPosts author (~210), getPost author (~184), getComments user (~253), getStories user (~340), getMessages sender (~427), getConversations json_agg (~413) — select explicit fields, NOT the whole row — getFollowers/getFollowing (~298/305), getFlashSales/getFlashSale artist, getBookings/getBooking artist+client, getJobs/getJobById studio, getStudioApprovalRequests artist+studio (~682), getLivestreamEvents host.
   - NEVER return hashedPassword or email to clients except email only to the authenticated owner via /api/users/me. The admin queries already select explicit columns — copy that style.
   - Verify: `grep -rn "hashedPassword" server/` and confirm it only appears in auth (bcrypt compare/hash) and the safeUser scrubber, never in a client response path.

2. Studio-approval IDOR. server/routes.ts PUT /api/studio-approvals/:id/approve and /reject (~1125-1141) only check requireRole(["STUDIO"]). Before mutating, load the request and return 403 unless request.studioId === req.userId. Add a getStudioApprovalRequestById(id) to server/storage.ts if needed.

3. Livestream start/end authorization. server/routes.ts POST /api/live-events/:eventId/start and /end (~944-966) have no ownership check. Load the event and return 403 unless event.hostId === req.userId (mirror the check already in PUT /api/livestream-events/:id at ~924).

4. Media-delete authorization is broken and insecure. server/routes.ts DELETE /api/upload/:publicId (~1053-1076). The generic prefixes nullify per-user checks, and real Cloudinary IDs are namespaced `inktagram/...` so legitimate deletes 403. Replace the string-prefix logic with real ownership: require the publicId to start with `inktagram/<folder>/<req.userId>/` OR verify against a DB record that the asset belongs to the user. Update server/services/cloudinary.ts uploadMedia to place uploads under `inktagram/${folder}/${userId}/...` so ownership is encoded, and pass userId through from the /api/upload handler.

5. Stop leaking internal error text. Across server/routes.ts handlers and the global handler in server/index.ts (~110-116): log the real error server-side, but return a generic message for 500s (keep specific 400/403/404 messages). Remove the `throw err;` after `res.status().json()` in index.ts (it throws after the response is sent).

6. Add server-side Zod validation where missing:
   - POST /api/flash-sales (~705): validate title, positive originalPriceCents/flashPriceCents (flash < original), availableSlots >= 1, expiresAt in the future.
   - POST /api/bookings (~760): validate required fields, scheduledAt must be in the future, depositCents <= totalPriceCents, non-negative amounts.
   - POST /api/jobs/:id/apply (~672): validate coverLetter; reject duplicate applications (same artistId+jobId) and applications to inactive jobs.
   Put schemas in server/utils/validation.ts alongside the others.
